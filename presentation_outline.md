# Pantry Path Presentation Outline

## Slide 1: Title and Overview

* Pantry Path
* Mervyn S. Philip
* A command-line meal planner that tracks pantry items, suggests recipes, and builds a shopping list.

Speaker note: Briefly explain that the project was selected because it gives a practical reason to use collections, loops, decisions, and input together.

## Slide 2: Live Demo Recap

* User views the pantry.
* User updates ingredient quantities.
* App suggests meals based on available ingredients.
* App creates a shopping list from missing recipe ingredients.

Speaker note: Connect the live demo to the required material: variables, strings, numbers, collections, control flow, loops, and user interaction.

## Slide 3: What Went Right

* The menu design made the program easy to test one feature at a time.
* The dictionary-based pantry kept ingredient quantity and category data organized.

Speaker note: Explain how small menu options helped development stay focused and how the data structure matched the real problem.

## Slide 4: What Needs Improvement

* The app does not save pantry changes after the program closes.
* Recipes are currently stored directly in the code instead of a separate file.

Speaker note: These are realistic next steps: file saving would improve usefulness, and external recipe data would make updates easier.

## Slide 5: Risk and Change Management

* Technical risk: using nested dictionaries could become confusing if the app grew too quickly.
* Management choice: keep the project small, test helper logic, and document why each collection type was used.

Speaker note: Mention that limiting scope protected the final demo from becoming unstable.

## Slide 6: Actionable Advice

* Build the smallest complete version first, then add polish.
* Test the logic that is hardest to check manually.

Speaker note: Close by connecting these lessons to future coding projects and the value of steady version control.
